# TappsMCP Feedback: Hooks Fail on Windows (PowerShell)

**Purpose:** Upstream feedback for TappsMCP so generated hooks work on Windows without local patches.

**Date:** 2026-02-24  
**Context:** HomeIQ project uses TappsMCP; `tapps_init` generates `.sh` hooks that do not run correctly on Windows when the host shell is PowerShell.

---

## Summary

PostToolUse / afterFileEdit–style hooks generated by TappsMCP are **bash-only**. On **Windows with PowerShell** they fail or never run, so the “run `tapps_quick_check` after editing a Python file” reminder does not appear.

---

## Environment

- **OS:** Windows (win32 10.x)
- **Shell:** PowerShell (default in Cursor/VS Code on Windows)
- **Hook paths:** `.claude/hooks/tapps-post-edit.sh`, `.cursor/hooks/tapps-after-edit.sh`
- **Trigger:** After Edit/Write tool use (Claude) or `afterFileEdit` (Cursor)

---

## Root Cause

The hook scripts assume a **Unix/bash** environment:

| Issue | Why it fails on Windows/PowerShell |
|-------|-------------------------------------|
| **Shebang / interpreter** | `#!/usr/bin/env bash` – PowerShell does not execute `.sh` files; they are not run by default. |
| **`python3`** | On Windows, the standard launcher is often `python` or `py`; `python3` may be missing from PATH. |
| **`grep -qE '\.py$'`** | `grep` is not a built-in PowerShell command; Unix utilities may be absent. |
| **`2>/dev/null`** | `/dev/null` is Unix-specific; PowerShell uses different redirection. |
| **`$(cat)` / stdin** | Bash `$(cat)` semantics differ from PowerShell; stdin handling can differ when the host invokes the hook. |

So the **root cause** is: **hooks are generated only as bash scripts and rely on Unix-only commands**, with no Windows/PowerShell alternative or detection.

---

## Suggested Fix (for TappsMCP)

1. **Ship both bash and PowerShell variants for hook scripts**
   - For each hook that reads stdin/JSON (e.g. PostToolUse/Edit, afterFileEdit, beforeMCPExecution, stop):
     - Keep the existing `.sh` for Unix/macOS/Git Bash.
     - Add a `.ps1` equivalent that uses only PowerShell and .NET (e.g. `ConvertFrom-Json`), no `python3`/`grep`.
   - In `tapps_init` (or equivalent), when writing hook config (e.g. Cursor `hooks.json`, Claude hook config):
     - **On Windows:** write the command to run the `.ps1` (e.g. `powershell -NoProfile -ExecutionPolicy Bypass -File .cursor/hooks/tapps-after-edit.ps1`).
     - **On Unix/macOS:** keep the current `.sh` command.

2. **Optional: platform detection in `tapps_init`**
   - Detect OS (e.g. `process.platform === 'win32'` or `os.name == 'nt'`) and:
     - Install/copy the `.ps1` hooks on Windows and point the hook config at them.
     - Install/copy the `.sh` hooks on Unix and point the hook config at them.

3. **Make bash scripts more portable where kept**
   - Prefer `python` with a fallback: `command -v python3 >/dev/null 2>&1 && python3 -c "..." || python -c "..."` so Git Bash on Windows still works if only `python` is in PATH.
   - Avoid relying on `grep` for simple checks (e.g. do “ends with .py” inside the same Python one-liner).

4. **Documentation**
   - In AGENTS.md or hook docs: note that on Windows, PowerShell hooks (`.ps1`) are used when generated; if users copy only the `.sh` from another machine, they may need to use the `.ps1` or run the `.sh` under bash (e.g. Git Bash).

---

## Workaround We Applied (HomeIQ)

- Added a local **PowerShell** implementation of the after-edit reminder:  
  `.cursor/hooks/tapps-after-edit.ps1`  
  - Reads stdin, parses JSON with `ConvertFrom-Json`, supports both `file` and `tool_input.file_path` / `path`.
- Updated **Cursor** `hooks.json` so `afterFileEdit` runs the `.ps1` via:  
  `powershell -NoProfile -ExecutionPolicy Bypass -File .cursor/hooks/tapps-after-edit.ps1`
- Hardened **.claude** `tapps-post-edit.sh` for bash-on-Windows: use `python3` or `python`, and do the “.py” check inside Python so `grep` is not required.

This fixes behavior on our side but duplicates logic and will drift from upstream if TappsMCP changes hook payload or behavior. **We’d prefer TappsMCP to ship and wire up .ps1 hooks on Windows so we can drop the local workaround.**

---

## Payload Shape (for parity)

- **Cursor `afterFileEdit`:** JSON has top-level `file` (path string).
- **Claude PostToolUse (Edit/Write):** JSON has `tool_input` with `file_path` or `path`.

A PowerShell hook that should work for both can do: use `file` if present, else `tool_input.file_path`, else `tool_input.path`, else `"unknown"`.
