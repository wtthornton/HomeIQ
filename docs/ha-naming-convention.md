# Home Assistant Naming Convention for HomeIQ AI Discovery

**Status:** Draft
**Applies to:** All Home Assistant entities, areas, labels, and aliases used with HomeIQ
**Goal:** Make entities reliably discoverable by AI without disrupting the HA user experience

---

## Design Principles

1. **User experience first** — Friendly names stay human-readable ("Kitchen Ceiling Light"). Users never see internal conventions in the HA dashboard.
2. **AI discovery via structure** — Entity IDs, areas, labels, and aliases follow predictable patterns so the AI resolves entities with high confidence instead of guessing.
3. **Opt-in enrichment** — Labels and aliases are additive. Nothing breaks if you skip them; the AI just gets less context.
4. **One source of truth** — Home Assistant's Entity Registry is the canonical source. HomeIQ syncs from it via websocket-ingestion and stores the result in the `entities` table (data-api).

---

## 1. Area Naming Standard

Areas are the primary filter for entity resolution. Consistent area names dramatically reduce false positives.

### Rules

| Rule | Example | Why |
|---|---|---|
| Use singular nouns | `Kitchen`, not `Kitchens` | Matches natural speech ("turn off the kitchen lights") |
| Title Case, no abbreviations | `Living Room`, not `LR` or `living_room` | HA normalizes to `area_id = living_room` automatically |
| No floor prefixes in area name | `Master Bedroom`, not `2F Master Bedroom` | Use HA's floor feature for floor grouping |
| Be specific for duplicate rooms | `Kids Bedroom`, `Guest Bedroom` | Avoids ambiguity when user says "bedroom lights" |
| Use HA's built-in floor feature | Floor: `Upstairs` → Area: `Master Bedroom` | AI can resolve "upstairs bedroom" via floor + area |

### Standard Area Names (Adapt to Your Home)

```
Kitchen              Living Room          Dining Room
Master Bedroom       Kids Bedroom         Guest Bedroom
Office               Bathroom             En Suite
Garage               Hallway              Landing
Utility Room         Garden               Patio
Basement             Attic                Nursery
```

### What HomeIQ Does With Areas

- `entity_resolution_service.py` extracts area names from user prompts and filters entities by `area_id`
- Area matching is case-insensitive with underscore/space normalization (`living room` = `living_room`)
- The hardcoded area list in `_extract_area_from_prompt()` will be replaced by dynamic area lookup from the entity cache (see Roadmap)

## 2. Entity ID Convention

Entity IDs are set by integrations and are generally not user-editable. However, when you **do** control them (custom entities, template sensors, helpers, MQTT devices), follow this pattern:

### Format

```
{domain}.{area}_{device_type}_{qualifier}
```

### Examples

| Entity ID | Breakdown |
|---|---|
| `light.kitchen_ceiling_main` | domain=light, area=kitchen, type=ceiling, qualifier=main |
| `light.office_desk_lamp` | domain=light, area=office, type=desk, qualifier=lamp |
| `sensor.garden_temperature` | domain=sensor, area=garden, type=temperature |
| `switch.garage_door_relay` | domain=switch, area=garage, type=door, qualifier=relay |
| `binary_sensor.hallway_motion` | domain=binary_sensor, area=hallway, type=motion |
| `climate.living_room_thermostat` | domain=climate, area=living_room, type=thermostat |
| `cover.bedroom_blinds_left` | domain=cover, area=bedroom, type=blinds, qualifier=left |
| `sensor.kitchen_power_total` | domain=sensor, area=kitchen, type=power, qualifier=total |

### Rules

| Rule | Good | Bad | Why |
|---|---|---|---|
| Area comes first after domain | `light.kitchen_ceiling` | `light.ceiling_kitchen` | Area is the primary filter — putting it first enables prefix matching |
| Use underscores, lowercase | `light.office_desk_lamp` | `light.Office-Desk-Lamp` | HA convention; consistent with `area_id` normalization |
| Be descriptive, not coded | `light.office_ceiling_left` | `light.office_hue_1_6` | AI can't infer position from Hue channel numbers |
| Include device class hint | `sensor.office_temperature` | `sensor.office_1` | Makes domain + intent clear without looking up attributes |
| Qualifier for disambiguation | `light.kitchen_counter_left` | `light.kitchen_counter_2` | Positional words ("left", "right", "main") beat numbers |

### Integration-Generated IDs

Many entity IDs are auto-generated by integrations (e.g., `light.hue_color_downlight_1_6`). You **cannot** rename these. Instead:
- Set a meaningful **friendly name** (Section 3)
- Add **aliases** for AI discovery (Section 5)
- The name enhancement system in `device-intelligence-service` can suggest better names automatically

## 3. Friendly Name Convention

Friendly names are what users see in the HA dashboard and what the AI uses for natural language matching.

### Format

```
{Area} {Position?} {Device Type} {Qualifier?}
```

### Examples

| Friendly Name | Entity ID |
|---|---|
| Kitchen Ceiling Light | `light.kitchen_ceiling_main` |
| Office Desk Lamp | `light.office_desk_lamp` |
| Garden Temperature | `sensor.garden_temperature` |
| Hallway Motion | `binary_sensor.hallway_motion` |
| Living Room Thermostat | `climate.living_room_thermostat` |
| Bedroom Left Blind | `cover.bedroom_blinds_left` |
| Kitchen Total Power | `sensor.kitchen_power_total` |

### Rules

| Rule | Good | Bad | Why |
|---|---|---|---|
| Start with area name | "Kitchen Ceiling Light" | "Ceiling Light Kitchen" | Matches how people talk ("kitchen lights") |
| Title Case | "Office Desk Lamp" | "office desk lamp" | Consistent display in HA UI |
| No manufacturer names | "Office Ceiling Light" | "Philips Hue Office Light" | AI doesn't need brand; clutters resolution |
| No model numbers | "Kitchen Sensor" | "Aqara RTCGQ11LM Kitchen" | Technical details belong in device info |
| No integration prefixes | "Office Temperature" | "Zigbee Office Temperature" | Protocol is irrelevant to the user/AI |
| Use natural position words | "Office Back Left Light" | "Office Light 6" | Humans and AI understand "back left", not "6" |

### Name Priority (How HomeIQ Resolves)

HomeIQ uses this priority chain from the entity model:

1. **`name_by_user`** — User's custom name in HA (highest priority)
2. **`name`** — Entity Registry name
3. **`original_name`** — Integration's default name (fallback)

The computed `friendly_name` field stores the winner. If you set `name_by_user` in HA, that's what HomeIQ sees.

## 4. Labels for AI Metadata

HA labels (2024.4+) are the primary mechanism for adding AI-consumable metadata without affecting the user interface. Labels are invisible in most dashboard views but are synced to HomeIQ's `entities.labels` column.

### HomeIQ Label Taxonomy

Use these label prefixes to tag entities for AI behavior:

#### Automation Intent Labels

| Label | Meaning | Example Use |
|---|---|---|
| `ai:automatable` | Entity can be automated by HomeIQ | Lights, switches, covers — things the AI should suggest automations for |
| `ai:monitor-only` | Entity should be read but never controlled | Utility meter, weather sensors — data sources only |
| `ai:ignore` | Entity should be excluded from AI processing | System entities, internal helpers, diagnostic sensors |
| `ai:critical` | Entity is safety-critical — require confirmation | Door locks, alarm panel, garage door |

#### Sensor Role Labels

| Label | Meaning | Example Use |
|---|---|---|
| `sensor:primary` | Primary sensor for its area/type | The "main" temperature sensor in a room (vs. a device's internal temp) |
| `sensor:trigger` | Commonly used as automation trigger | Motion sensor, door contact, presence detector |
| `sensor:condition` | Commonly used as automation condition | Lux sensor, time-of-day helper, occupancy |
| `sensor:diagnostic` | Device health / diagnostic data | Battery level, signal strength, uptime |

#### Grouping Labels

| Label | Meaning | Example Use |
|---|---|---|
| `group:all-lights` | Part of "all lights" group | Every controllable light |
| `group:night-lights` | Part of nighttime lighting | Dim hallway lights, bathroom night light |
| `group:media` | Part of media system | TV, soundbar, receiver, streaming devices |
| `group:security` | Part of security system | Cameras, locks, alarm, motion sensors |
| `group:climate` | Part of climate control | Thermostats, TRVs, fans, humidifiers |

#### Energy Labels

| Label | Meaning | Example Use |
|---|---|---|
| `energy:producer` | Generates energy | Solar panels, battery discharge |
| `energy:consumer` | Consumes energy | Washer, oven, EV charger |
| `energy:meter` | Measures energy flow | Smart meter, CT clamp, plug meter |
| `energy:tariff` | Tariff/pricing entity | Octopus rate, time-of-use helper |

### How to Apply Labels in HA

1. **Settings → Devices & Services → Entities** tab
2. Select entities → **Add Label**
3. Create labels using the prefixes above (HA will auto-create them)

Or via `configuration.yaml` for bulk setup:
```yaml
homeassistant:
  customize:
    light.kitchen_ceiling_main:
      # Labels are managed in the Entity Registry UI, not YAML
      # Use the HA UI or REST API to add labels
```

> **Note:** Labels are managed through HA's Entity Registry UI or REST API, not YAML customization. The `customize` block only handles attributes like `friendly_name` and `icon`.

## 5. Aliases for Natural Language

Aliases (HA 2023.8+) let you add alternative names that the AI and voice assistants use for matching, without changing the displayed friendly name.

### When to Add Aliases

- When users refer to an entity by a different name than its friendly name
- When there are common abbreviations or slang for a device
- When the entity serves multiple conceptual roles

### Examples

| Entity | Friendly Name | Aliases |
|---|---|---|
| `light.kitchen_ceiling_main` | Kitchen Ceiling Light | `kitchen lights`, `kitchen light`, `the big light` |
| `light.office_desk_lamp` | Office Desk Lamp | `desk light`, `my lamp`, `office lamp` |
| `climate.living_room_thermostat` | Living Room Thermostat | `heating`, `the thermostat`, `living room heating` |
| `cover.bedroom_blinds_left` | Bedroom Left Blind | `bedroom blinds`, `left blind`, `bedroom curtains` |
| `switch.garden_irrigation` | Garden Irrigation | `sprinklers`, `watering`, `garden water` |
| `media_player.living_room_tv` | Living Room TV | `the telly`, `TV`, `living room telly` |

### Rules

| Rule | Good Alias | Bad Alias | Why |
|---|---|---|---|
| Use how people actually talk | `the big light` | `primary ceiling luminaire` | Match real speech patterns |
| Include singular and plural | `kitchen light`, `kitchen lights` | Only one form | Users say both |
| Include common abbreviations | `TV`, `AC` | Only full name | Casual speech uses abbreviations |
| Include area-less references when unambiguous | `the thermostat` (if only one) | — | Users drop area when context is obvious |
| Skip aliases that match friendly name | — | Same as friendly name | Redundant; already matched |

### How to Add Aliases in HA

1. **Settings → Devices & Services → Entities** tab
2. Click an entity → **Aliases** section → Add aliases
3. Or via HA REST API: `POST /api/config/entity_registry/{entity_id}` with `aliases` array

## 6. Device Class Best Practices

Device classes help the AI understand what a sensor measures or what a device does without parsing the entity name.

### Ensure These Are Set Correctly

| Domain | Device Class | What It Tells AI |
|---|---|---|
| `binary_sensor` | `motion` | Trigger for presence-based automations |
| `binary_sensor` | `door` | Security / entry monitoring |
| `binary_sensor` | `window` | Security / ventilation |
| `binary_sensor` | `occupancy` | Room presence (different from motion) |
| `sensor` | `temperature` | Climate condition |
| `sensor` | `humidity` | Climate condition |
| `sensor` | `illuminance` | Lighting condition |
| `sensor` | `power` | Energy monitoring (watts) |
| `sensor` | `energy` | Energy monitoring (kWh) |
| `sensor` | `battery` | Device health |
| `cover` | `blind` | Window covering |
| `cover` | `garage_door` | Security-critical cover |

### How HomeIQ Uses Device Class

- Entity resolution filters by `device_class` when the user mentions a concept ("motion sensors", "temperature")
- Energy analytics uses `power`/`energy` device classes to auto-discover metering entities
- The entity model stores `device_class` with an index for fast lookups

## 7. Entities to Exclude (Blacklist Guidance)

Not every HA entity should be visible to the AI. Use the blacklist system (Epic 25.2) or `ai:ignore` labels to hide:

### Always Exclude

| Pattern | Reason |
|---|---|
| `sensor.*_battery` (unless you want battery monitoring) | Noise — dozens of battery sensors |
| `sensor.*_signal_strength` | Diagnostic only |
| `sensor.*_linkquality` | Diagnostic only |
| `update.*` | Firmware updates — not automatable |
| `button.*_identify` | Integration utility buttons |
| `number.*_calibration_*` | Device configuration, not user-facing |
| `sensor.hacs*` | HACS internal state |
| `persistent_notification.*` | HA system notifications |

### Consider Excluding

| Pattern | When to Exclude |
|---|---|
| `binary_sensor.*_tamper` | Unless security monitoring is active |
| `sensor.*_uptime` | Unless device health monitoring is wanted |
| `switch.*_child_lock` | Unless you automate child locks |
| `cover.*` | If you never want AI to control covers (safety) |

### Blacklist Configuration

Blacklists are managed via admin-api (Story 25.2):
- **Glob patterns:** `sensor.*_battery`, `update.*`
- **Exact IDs:** `switch.computer1`
- **Area-based:** `area:Garage` (exclude everything in garage from AI)

## 8. Naming Cleanup Checklist

Use this checklist to audit and clean up your HA install for HomeIQ:

### Step 1: Areas (5 minutes)
- [ ] Every device is assigned to an area
- [ ] Area names follow the standard (Title Case, singular, no abbreviations)
- [ ] No duplicate/ambiguous area names (e.g., two areas both called "Bedroom")
- [ ] Floors are set (HA 2024.1+) for multi-story homes

### Step 2: Friendly Names (15-30 minutes)
- [ ] Every user-facing entity has a descriptive friendly name
- [ ] Names follow `{Area} {Device Type} {Qualifier}` pattern
- [ ] No manufacturer names or model numbers in friendly names
- [ ] No integration prefixes (Zigbee, Z-Wave, etc.)
- [ ] Position words used instead of numbers ("Left" not "2")
- [ ] Run device-intelligence-service's name enhancement for AI suggestions

### Step 3: Labels (10-20 minutes)
- [ ] Controllable entities labeled `ai:automatable`
- [ ] Read-only sensors labeled `ai:monitor-only`
- [ ] Safety-critical entities labeled `ai:critical`
- [ ] Noisy/internal entities labeled `ai:ignore`
- [ ] Primary sensors labeled `sensor:primary`
- [ ] Common triggers labeled `sensor:trigger`

### Step 4: Aliases (10-15 minutes)
- [ ] High-use entities have natural language aliases
- [ ] Both singular and plural forms added
- [ ] Common abbreviations added (TV, AC, etc.)
- [ ] Casual/slang names added where applicable

### Step 5: Exclusions (5 minutes)
- [ ] Diagnostic sensors excluded (battery, signal, link quality)
- [ ] System entities excluded (HACS, updates, persistent notifications)
- [ ] Blacklist patterns configured in admin-api

### Step 6: Verify (5 minutes)
- [ ] Run entity discovery script (`scripts/standalone-entity-discovery.py`)
- [ ] Check data-api entity list matches expectations
- [ ] Test a few natural language commands through ha-ai-agent-service

## 9. How HomeIQ Uses Each Field

| Field | Stored In | Used By | Purpose |
|---|---|---|---|
| `entity_id` | `entities.entity_id` | Entity resolver (exact match, tier 1) | Primary identifier |
| `friendly_name` | `entities.friendly_name` | Entity resolver (name match, tier 2-3), UI display | Human-readable name for matching and display |
| `name_by_user` | `entities.name_by_user` | Friendly name computation (highest priority) | User's custom name override |
| `aliases` | `entities.aliases` (JSONB) | Entity resolution scoring (`_score_entities`) | Alternative names for fuzzy matching |
| `labels` | `entities.labels` (JSONB) | AI intent filtering, energy discovery, blacklisting | Metadata tags for AI behavior |
| `area_id` | `entities.area_id` | Area-based entity filtering (primary filter) | Room/location grouping |
| `domain` | `entities.domain` | Domain filtering (`light`, `sensor`, etc.) | Entity type classification |
| `device_class` | `entities.device_class` | Sensor type inference, energy auto-discovery | What the entity measures/does |
| `capabilities` | `entities.capabilities` (JSONB) | Service call validation (brightness, color, etc.) | What actions are supported |
| `available_services` | `entities.available_services` (JSONB) | Tool call routing | Which HA services can be called |

These improvements will make the naming convention self-enforcing:

- [ ] **HA setup wizard** — Health dashboard page that audits HA entities against this convention and guides cleanup
- [ ] **Auto-alias generation** — Generate alias suggestions from friendly names (singular/plural, abbreviations)

- [ ] **Convention compliance dashboard** — Show percentage of entities following the convention with per-entity recommendations
- [ ] **HA integration** — Custom HA integration that enforces naming convention on entity creation
- [ ] **Multi-home support** — Convention works across multiple HA instances with namespace prefixes
